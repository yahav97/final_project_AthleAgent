# שיקולי מודל למידת מכונה — AthleAgent

## 1. מדדי הערכה — מה כל מדד אומר?

### Recall (רגישות) — "כמה פציעות תפסנו?"

```
Recall = פציעות שזוהו נכון / סך כל הפציעות האמיתיות
```

**דוגמה:** מתוך 100 פציעות אמיתיות, המודל זיהה 87 → Recall = 87%

**למה זה הכי חשוב אצלנו?**
פציעה שלא זוהתה (False Negative) מסוכנת — הספורטאי ממשיך להתאמן ומסתכן בפציעה חמורה.
התראת שווא (False Positive) פחות מסוכנת — הספורטאי מקבל המלצה לנוח, במקרה הרע מנוחה מיותרת.

**→ אנחנו מעדיפים לטעות בכיוון של "יותר מדי אזהרות" מאשר לפספס פציעה.**

### Precision (דיוק חיובי) — "כמה מהתראות באמת פציעה?"

```
Precision = פציעות שזוהו נכון / סך כל ההתראות שהמודל נתן
```

**דוגמה:** המודל נתן 350 התראות, מתוכן 49 היו פציעות אמיתיות → Precision = 14%

**למה הוא נמוך אצלנו?**
כי שיעור הפציעות הוא ~8–12% מהימים. כדי לתפוס 87% מהפציעות, המודל חייב "לחשוד" ביותר ימים, מה שיוצר הרבה התראות שווא. זה Trade-off מכוון — ראו הסבר בהמשך.

### FPR (False Positive Rate) — "כמה ימים בריאים סימנו בטעות?"

```
FPR = ימים בריאים שסומנו כסיכון / סך כל הימים הבריאים
```

**סף מקסימלי:** 70%. אם יותר מ-70% מהימים הבריאים מקבלים התראה — המודל "צועק זאב" יותר מדי ולא שימושי.

### ROC-AUC — "כמה טוב המודל מפריד בין פצועים לבריאים?"

ערך בין 0.5 (מטבע) ל-1.0 (מושלם). מודד את **היכולת הכללית** של המודל לדרג נכון — ללא תלות בסף ספציפי.

| ערך | משמעות |
|---|---|
| 0.50 | ניחוש אקראי |
| 0.60–0.70 | בינוני |
| 0.70–0.80 | טוב |
| 0.80+ | מצוין |

**אצלנו: 0.723** — טוב.

### Brier Score — "כמה מכוילות ההסתברויות?"

מודד את המרחק הריבועי הממוצע בין ההסתברות שהמודל נתן לבין המציאות (0 או 1). **ככל שנמוך יותר — יותר טוב.**

| ערך | משמעות |
|---|---|
| 0.00 | מושלם |
| 0.10 | מצוין |
| 0.15 | טוב |
| 0.25 | בינוני |

**אצלנו: 0.115** — מצוין.

### LogLoss — "כמה 'בטוח' המודל כשהוא טועה?"

עונש לוגריתמי על הסתברויות שגויות. מודל שנותן 90% סיכון לספורטאי שלא נפצע מקבל עונש כבד. **ככל שנמוך יותר — יותר טוב.**

**אצלנו: 0.388** — טוב (בהשוואה ל-0.587 בגרסה הקודמת).

## 2. מדיניות הסף — מה ה-Minimum לכל מדד?

| מדד | סף מינימלי | סיבה |
|---|---|---|
| **Recall** | ≥ 80% (hard min) | אסור לפספס יותר מ-1 מתוך 5 פציעות |
| **Recall** | ≥ 85% (target) | יעד רגיל — לפספס מקסימום 1 מ-7 |
| **Recall** | ≥ 88% (high target) | יעד אידיאלי |
| **Precision** | ≥ 13% | מעל הבסיס האקראי (~10% שיעור פציעה) |
| **F1** | ≥ 0.22 | איזון מינימלי recall-precision |
| **FPR** | ≤ 70% | לא להציף את הספורטאי בהתראות |

### למה Recall ≥ 80% ולא 95%?

| Recall | Precision | FPR | מה קורה |
|---|---|---|---|
| 95% | ~10% | ~90% | כמעט כל יום נראה מסוכן — חסר תועלת |
| 90% | ~13% | ~75% | הרבה התראות, אבל תופס כמעט הכל |
| **87%** | **14%** | **65%** | **נקודת עבודה נוכחית — איזון טוב** |
| 80% | ~18% | ~55% | פחות התראות, אבל מפספס 1 מ-5 |
| 70% | ~25% | ~40% | דיוק טוב יותר, אבל מפספס 3 מ-10 |

**בחרנו Recall=87% כי:**
- תופס את רוב הפציעות (87 מתוך 100)
- ה-FPR (65%) עדיין סביר — לא כל יום מוצג כ-"High Risk"
- כשמשלבים עם 3 רמות סיכון (Low/Medium/High), הספורטאי מקבל תמונה מפורטת

## 3. בחירת מודל — למה XGBoostDeep?

### המודלים שהשוו (11 מודלים)

| משפחה | מודלים | סוג |
|---|---|---|
| Logistic Regression | 1 | ליניארי, בסיס |
| Random Forest | 2 (רגיל + tuned) | Ensemble של עצי החלטה |
| Extra Trees | 2 (רגיל + tuned) | כמו RF אבל עם פיצולים אקראיים |
| Gradient Boosting | 1 | Boosting קלאסי |
| XGBoost | 5 גרסאות | Boosting מתקדם (Raw, Deep, Calibrated, tuned) |

### תוצאות ההשוואה — Run `20260512_075115`

| מודל | Recall | Precision | F1 | FPR | ROC-AUC | Brier |
|---|---|---|---|---|---|---|
| ExtraTrees | 81.3% | 15.0% | 0.253 | 56.9% | 0.711 | 0.200 |
| LogisticRegression | 79.6% | 14.9% | 0.251 | 56.1% | 0.704 | 0.214 |
| RandomForest | 71.2% | 17.6% | 0.282 | 41.1% | 0.720 | 0.182 |
| **XGBoostDeep** | **86.6%** | **14.1%** | **0.242** | **65.1%** | **0.723** | **0.115** |
| XGBoostRaw | 88.9%* | 13.7% | 0.237 | 69.1% | 0.725 | 0.097 |
| ExtraTreesTuned | 91.1%* | 13.1% | 0.228 | 74.7%** | 0.716 | 0.102 |

*\* הערכים שמסומנים ב-\* הם בנקודות עבודה אחרות (best operating point), לא ב-threshold=0.4*
*\*\* FPR > 70% = חורג ממגבלת FPR*

### למה XGBoostDeep נבחר ולא XGBoostRaw?

תהליך הבחירה (`pick_best_model`) עובד ב-3 רמות (Tiers):

**Tier 0 (עדיף):** עומד בכל הדרישות
- Recall ≥ 85%, FPR ≤ 70%, Precision ≥ 13%, F1 ≥ 0.22

**Tier 1:** עומד במינימום
- Recall ≥ 80%, FPR ≤ 70%

**Tier 2:** לא עומד — fallback

**בתוך כל Tier, הסדר הוא:**
1. FPR נמוך ביותר (פחות התראות שווא)
2. Recall גבוה ביותר
3. Precision גבוה ביותר
4. F1 גבוה ביותר
5. ROC-AUC גבוה ביותר

**XGBoostDeep נבחר כי:**
- הוא ב-**Tier 0** — עומד בכל הדרישות (Recall=86.6%, FPR=65.1%)
- ה-FPR שלו (65.1%) **נמוך יותר** מ-XGBoostRaw (69.1%) — כלומר פחות התראות שווא
- XGBoostRaw אמנם בעל Recall קצת יותר גבוה (88.9%), אבל ה-FPR שלו קרוב מאוד למגבלה (70%)
- ExtraTreesTuned חורג מ-FPR ≤ 70% ולכן נפסל

### למה XGBoost ולא Random Forest?

| יתרון XGBoost | הסבר |
|---|---|
| Boosting vs Bagging | XGBoost בונה עצים ברצף, כל עץ מתקן טעויות של הקודם. RF בונה עצים במקביל |
| רגולריזציה מובנית | `reg_alpha`, `reg_lambda`, `gamma` מונעים overfitting |
| `scale_pos_weight` | מטפל בחוסר איזון (8% פציעות) באופן אלגנטי |
| מהירות | מהיר יותר ב-inference (חשוב לשרת production) |

### פרמטרי המודל הנבחר (XGBoostDeep)

| פרמטר | ערך | תפקיד |
|---|---|---|
| `n_estimators` | 500 | מספר עצים |
| `max_depth` | 7 | עומק מקסימלי לעץ |
| `learning_rate` | 0.03 | קצב למידה (נמוך = יותר עצים, יציבות) |
| `subsample` | 0.85 | % דוגמאות לכל עץ (מניעת overfitting) |
| `colsample_bytree` | 0.8 | % פיצ'רים לכל עץ |
| `colsample_bylevel` | 0.8 | % פיצ'רים לכל רמה בעץ |
| `reg_alpha` | 0.5 | L1 רגולריזציה |
| `reg_lambda` | 2.0 | L2 רגולריזציה |
| `min_child_weight` | 5 | מינימום דוגמאות בעלה |
| `gamma` | 0.1 | מינימום Gain לפיצול |
| `scale_pos_weight` | 3.5 | משקל יתר לדוגמאות פציעה |

## 4. ה-Trade-off המרכזי: Recall vs Precision

### המצב שלנו

```
    Recall גבוה                              Precision גבוה
    (תופסים הכל)                             (מדייקים)
    ←─────────────────────X──────────────────────→
                    נקודת העבודה שלנו
                    Recall=87%, Precision=14%
```

### למה לא פשוט להעלות את שניהם?

בבעיה עם **8% שיעור פציעה**, יש מגבלה מתמטית:
- ב-100 ימים, ~8 ימי פציעה ו-~92 ימים בריאים
- כדי לתפוס 7 מתוך 8 פציעות (Recall=87%), צריך להרחיב את "רשת" ההתראות
- הרשת הרחבה תופסת גם ~60 ימים בריאים בטעות
- אז מתוך ~67 התראות, רק 7 אמיתיות = Precision ~10%

**זה לא באג — זה מגבלה מובנית של שיעור פציעה נמוך.**

### איך זה עובד בפרקטיקה — 3 רמות סיכון

המודל לא רק אומר "כן/לא". הוא נותן **ציון סיכון רציף (0–100%)** שמתורגם ל-3 רמות:

| רמת סיכון | ציון | שיעור פציעה אמיתי | % מהימים | מה הספורטאי רואה |
|---|---|---|---|---|
| **Low** (ירוק) | < 11% | 5.0% | 42% | "הכל בסדר, המשך כרגיל" |
| **Medium** (צהוב) | 11%–18% | 11.4% | 49% | "שים לב, יש סימנים — שקול להפחית עומס" |
| **High** (אדום) | ≥ 18% | 37.2% | 9% | "סיכון גבוה — מומלץ לנוח" |

**בזון האדום, 37% מהימים באמת מסתיימים בפציעה** — זה שיעור גבוה מאוד (פי 4 מהממוצע).

## 5. חלוקת הנתונים (Train/Test Split)

| פרט | ערך |
|---|---|
| סך שורות | 345,000 |
| שיטת חלוקה | GroupShuffleSplit לפי `athlete_id` |
| יחס | 80% אימון, 20% טסט |
| Benchmark holdout | קבוצת אתלטים קבועה שנפרדה מראש |

**למה GroupShuffleSplit?**
כדי שספורטאי לא יופיע גם באימון וגם בטסט. אחרת המודל "שולף" מידע על הספורטאי ממוצא האימון → תוצאות אופטימיות מדי.

## 6. השוואה בין גרסאות

| מדד | v1 (36 פיצ'רים, XGBoostRaw) | v2 (34 פיצ'רים, XGBoostDeep) | שינוי |
|---|---|---|---|
| Recall | 90.3% | 86.6% | ↓ 3.7% |
| Precision | 27.4% | 14.1% | ↓ |
| ROC-AUC | 0.654 | **0.723** | ↑ 10.6% |
| Brier Score | 0.200 | **0.115** | ↑↑ (שיפור גדול) |
| LogLoss | 0.587 | **0.388** | ↑↑ (שיפור גדול) |
| FPR | 79.0% | **65.1%** | ↑↑ (פחות שווא) |
| Threshold | 0.30 | 0.18 | — |

**מה השתפר:**
- ROC-AUC עלה → המודל מפריד טוב יותר בין פצועים לבריאים
- Brier ו-LogLoss ירדו → ההסתברויות מכוילות יותר (כשהמודל אומר 30% → באמת ~30%)
- FPR ירד מ-79% ל-65% → פחות "צעקות זאב"

**מה ירד:**
- Recall ירד מ-90.3% ל-86.6% — עדיין מעל היעד (85%)
- Precision ירד — בגלל ה-threshold הנמוך יותר (0.18 במקום 0.30)
