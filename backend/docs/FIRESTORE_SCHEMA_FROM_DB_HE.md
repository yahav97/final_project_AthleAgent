# מבנה Firestore — תיעוד מהדאטה בפועל (פרויקט `athleagent`)

מסמך זה נוצר **רק** מתוך קריאה ישירה ל-Firestore (Firebase Admin), באמצעות סריקה של **כל** מסמכי המשתמשים הקיימים ושל **כל** מסמכי הקבוצות, ואיחוד שמות השדות והטיפוסים שנצפו. אין כאן הסקות ממסכי אפליקציה שלא הופיעו בדאטה.

| פרט | ערך |
|-----|-----|
| מזהה פרויקט (מה-Service Account) | `athleagent` |
| תאריך יצירת התיעוד | 2026-05-10 |
| שיטה | איחוד שדות על כל המסמכים תחת `users` ו-`teams`; תת-אוסף `meals` נסרק עד 50 מסמכים לכל יום תזונה (בפועל מספר קטן). |

**מה לא נכלל:** אוספים שאינם `users` או `teams` (לא נמצאו במסד הנוכחי). שדות שייכתבו בעתיד על ידי הבקאנד (למשל אחרי חיזוי) ועדיין לא הופיעו — מצוינים בסוף כ"לא נצפה".

---

## 1. אוספים ברמה העליונה

| אוסף | תיאור |
|------|--------|
| `users` | מסמך לכל משתמש; מזהה המסמך = `uid` של Firebase Auth |
| `teams` | מסמך לכל קבוצה; מזהה מסמך = מזהה אוטומטי של Firestore |

---

## 2. `users/{uid}` — פרופיל

מזהה המסמך: מחרוזת ה-UID של Auth.

### שדות שנצפו בכל מסמכי הפרופיל בפרויקט

| שדה | טיפוסים שנצפו | הערות |
|-----|----------------|--------|
| `uid` | מחרוזת | |
| `fullName` | מחרוזת | |
| `email` | מחרוזת | |
| `role` | מחרוזת | ערכים שנצפו כוללים `Athlete`, `Coach` |
| `teamId` | מחרוזת | מופיע כשיש שיוך לקבוצה; במסמכים של משתמש מסוג Coach השדה **לא** הופיע בדגימה |

### שדות שלא נצפו במסמכי `users/{uid}` בפרויקט הנוכחי

הבקאנד/חוזה עשויים לצפות לשדות פרופיל נוספים (למשל `age`, `historyInjuryCount`) — **אף אחד מהם לא הופיע** במסמכים שנסרקו. אם יתווספו בעתיד, יש לעדכן מסמך זה אחרי סריקה חוזרת.

---

## 3. `users/{uid}/daily_health/{yyyy-MM-dd}`

מזהה המסמך: תאריך בפורמט `yyyy-MM-dd`.

המסמך משמש לאחסון נתוני בריאות/סנכרון **ולשדות נגזרים** (ציון סיכון, המלצת AI). תוכן המסמך משתנה בין ימים — יש מסמכים עם שדה יחיד ויש מסמכים מלאים.

### איחוד כל השדות שנצפו בכל ימי ה-`daily_health` בפרויקט

| שדה | טיפוסים שנצפו | הערות |
|-----|----------------|--------|
| `sleepMinutes` | שלם | |
| `steps` | שלם | |
| `distanceMeters` | שלם | |
| `activeCalories` | שלם | |
| `totalCalories` | שלם | |
| `bmrCalories` | שלם | |
| `heartRateAvg` | שלם | |
| `heartRateMax` | שלם | |
| `heartRateMin` | שלם | |
| `avgHeartRate` | שלם | הופיע במסמך אחד לפחות; שונה משם `heartRateAvg` |
| `weightKg` | שלם או `float` | שני טיפוסים שונים בדאטה |
| `lastSync` | חותמת זמן Firestore (`DatetimeWithNanoseconds`) | |
| `finalRiskScore` | שלם | |
| `aiRecommendation` | מחרוזת | |
| `lastCheckInTime` | חותמת זמן | הופיע במסמך לפחות אחד תחת `daily_health` (מיזוג/shape לא אחיד) |

### שדות שתועדו בחוזה הבקאנד אך **לא נצפו** בדאטה הנוכחית

עדיין לא הופיעו במסמכים שנסרקו: `riskLevel`, `dataQualityScore`, `dataQualityStatus`, `predictionUpdatedAt`, `backendRecommendation` — ככל הנראה טרם נכתבו על ידי זרימת השרת או לא נשמרו בימים שנדגמו.

## 9. סיכום והבחנות לעבודה המשותפת

1. **עקביות שמות:** מופיעים גם `heartRateAvg` וגם `avgHeartRate` תחת `daily_health` — כדאי לאחד קונבנציה בצד הכתיבה (אפליקציה/סנכרון).
2. **`weightKg`:** קיים גם כשלם וגם כנקודה צפה — לקוחות יצירת דוחות צריכים לטפל בשני המקרים.
3. **תזונה:** סכומים ו-`mealsLoggedCount` נשמרים כ-`float` (טיפוס נפוץ אחרי `increment`).
4. **פרופיל למודל:** שדות כמו גיל או היסטוריית פציעות עשויים להיעדר ב-Firestore — הבקאנד משלים ברירות מחדל.
5. **עדכון מסמך:** אחרי שינוי משמעותי בדאטה או אחרי הפעלת כתיבת תוצאות חיזוי מהשרת, להריץ שוב סקריפט אינטגרציה / סריקה ולעדכן קובץ זה.
